Browse pour le Product Owner
Superpouvoir PO : vérifier le produit, capturer des preuves visuelles, tester des flows utilisateurs — sans quitter le terminal ni demander à un dev.
Vérifier une page
goto <url> — naviguer vers une page
text — lire le contenu texte nettoyé
links — lister tous les liens (broken links?)
screenshot [path] — capturer la page entière
screenshot --viewport [path] — capturer le viewport only
screenshot "#selector" [path] — capturer un composant
Tester un flow utilisateur
snapshot -i — lister tous les éléments interactifs (@e1, @e2…)
click @e3 — cliquer un bouton/lien par ref
fill @e5 "valeur" — remplir un champ
select @e7 "option" — choisir dans un dropdown
press Enter — simuler une touche
scroll down [5] — scroller la page
wait ".spinner" — attendre qu'un élément apparaisse
Vérifier la qualité
console --errors — erreurs JS visibles par l'utilisateur ?
is visible ".success-msg" — un élément est-il affiché ?
is enabled "#submit-btn" — le bouton est-il actif ?
perf — métriques de performance de la page
forms — structure JSON de tous les formulaires
accessibility — arbre ARIA (audit a11y rapide)
Comparer & Responsive
responsive [prefix] — 3 screenshots : mobile / tablet / desktop
diff <staging> <prod> — diff texte entre environnements
viewport 375 812 — simuler iPhone · viewport 1440 900 — desktop
Onglets & Navigation
back · forward · reload · url
tabs · tab <idx> · newtab · closetab <idx>
Snapshot — Le système de refs
snapshot -i parse l'accessibility tree et assigne des refs (@e1, @e2…) à chaque élément interactif. Plus besoin de connaître les sélecteurs CSS.
FlagQuand l'utiliser
-iLister boutons, liens, inputs → le plus utilisé
-aScreenshot annotée avec labels sur chaque élément
-DVoir ce qui a changé depuis le dernier snapshot
-s selZoomer sur une section (ex: -s ".sidebar")
-cVue compacte, masquer les éléments vides
-o pathSauvegarder screenshot annotée sur disque
Workflows Product Owner
1. Cadrer une feature (avant le dev)
/office-hours → Diagnostic produit, 6 questions
/plan-ceo-review → Challenger le scope, trouver le 10★
/plan-design-review → Valider l'UX en 7 passes
ou en 1 commande : /autoplan
2. Prototyper & itérer
/design-consultation → Design system
/design-shotgun → Explorer N variantes visuelles
/design-html → Prototype HTML interactif, live reload
/browse → Vérifier le proto dans le navigateur
3. Valider avant release
/qa → Test exhaustif + score santé (visual, UX, a11y, perf)
/benchmark → Core Web Vitals avant/après
/browse → responsive /tmp/check → vérifier 3 breakpoints
/cso → Audit sécu avant mise en prod
4. Après le deploy
/canary → Monitoring visuel + console + perf
/browse → diff staging prod → vérifier la parité
/retro → Rétro hebdo, métriques par personne

⚠ À retenir
• snapshot -i en premier — ne jamais deviner les sélecteurs
• Refs invalidées après navigation → refaire snapshot
• /qa quick = smoke test rapide, exhaustive = audit complet
